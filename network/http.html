                # HTTP/1.1

                • 高延迟

                由于队头阻塞(指当顺序发送的请求序列中的一个请求因为某种原因被阻塞时，后面排队的所有请求也一并被阻塞)，导致带宽无法被利用

                针对性的优化方向：

                • 同一页面的资源分散到不同域名下，提升连接上限，浏览器对于同一域名允许同时建立的 TCP 连接有上限
                • 合并小文件较少资源文件数量，例如雪碧图
                • 内联资源，例如将小图片转化成base64字符串
                • 减少请求数量，例如使用打包工具合并js文件
                • 无状态

                协议对于连接状态没有记忆功能，报文Header需要携带许多重复的"User Agent"、"Cookie"、"Accept"、"Server"等字段

                • 明文传输

                • 不支持服务器推送

                # HTTP/2

                基于SPDY协议，专注于性能，目标是在用户和网站间只用一个连接

                # 新特性

                • 二进制传输
                • 将请求和响应数据分割为更小的帧，并且采用二进制编码，以流的形式传输
                • 将原来的"Header+Body"的消息"打散"为数个小片的二进制"帧"(Frame)，用"HEADERS"帧存放头数据、"DATA"帧存放实体数据
                • Header压缩

                使用HPACK压缩算法，在客户端和服务器两端建立“字典”，用索引号表示重复的字符串，还采用哈夫曼编码来压缩整数和字符串，可以达到50%~90%的高压缩率

                • 在客户端和服务器端使用“首部表”来跟踪和存储之前发送的键-值对，对于相同的数据，不再通过每次请求和响应发送
                • 首部表在HTTP/2的连接存续期内始终存在，由客户端和服务器共同渐进地更新
                • 每个新的首部键-值对要么被追加到当前表的末尾，要么替换表中之前的值
                • 多路复用
                • 同域名下所有通信都在单个连接上完成
                • 单个连接可以承载任意数量的双向数据流
                • 数据流以消息的形式发送，而消息又由一个或多个帧组成，多个帧之间可以乱序发送，因为根据帧首部的流标识可以重新组装。
                • 服务器推送

                浏览器请求HTML的时候就提前把可能会用到的JS、CSS文件发给客户端，减少等待的延迟

                • 安全性

                基于 TLS 加密，一般都是 HTTPS

                # 缺点

                • 建连延时高

                TCP和TLS建立连接都需要握手延迟过程，传输数据前需要额外花费3~4 RTT

                • 队头阻塞没有彻底解决

                丢包重连机制和单个 TCP 连接当网络较差时，请求延迟会更严重

                • 多路复用导致服务器压力上升

                不限制同时请求数量，会导致出现瞬时QPS暴涨

                • 多路复用容易Timeout

                # HTTP/3

                基于QUIC协议，在传输层用UDP协议代替TCP协议，通过融合传输层、表示层，既缩短了握手时长，也加密了传输层中的绝大部分字段，提升了网络安全性

                • Packet Header 实现了可靠的连接。当UDP报文丢失后，通过 Packet Header 中的 Packet Number 实现报文重传。连接也是通过其中的 Connection ID 字段定义的
                  • Long Packet Header 用于首次建立连接
                  • Short Packet Header 用于日常传输数据
                • QUIC Frame Header 在无序的 Packet 报文中，基于 QUIC Stream 概念实现了有序的字节流，这允许 HTTP 消息可以像在TCP连接上一样传输
                • HTTP3 Frame Header 定义了 HTTP Header、Body 的格式，以及服务器推送、QPACK编解码流等功能。

                # 新特性

                • 类似 TCP 的流量控制、传输可靠性
                • 可插拔，应用程序层面可以实现不同的拥塞控制算法
                • 单调递增的 Packet Number，使用 Packet Number 代替了 TCP 的 Seq
                • 不允许 Reneging， 一个 Packet 只要被 ACK，就认为它一定被正确接收
                • 前向纠错（FEC）
                • 更多的 ACK 块和增加 ACK Delay 时间
                • 基于 Stream 和 Connection 级别的流量控制
                • 快速握手

                使用0-RTT或者1-RTT来建立连接

                • 集成 TLS 加密算法
                • 基于TLS1.3，完全握手情况下，需要 1-RTT 建立连接，恢复会话可以直接发送加密后的应用数据，不需要额外的 TLS 握手，也就是 0-RTT
                • 无法保证前向安全，一旦得到Session Ticket Key就可以解密之前的数据，可以设置相关参数超时缓解问题
                • 多路复用（彻底解决TCP中队头阻塞的问题）

                QPACK压缩算法使用独立的单向 Stream 分别传输动态表编码、解码信息，这样乱序、并发传输HTTP消息的 Stream 既不会出现队头阻塞，也能基于时序性大幅压缩 HTTP Header 的体积

                • 连接迁移
                • QUIC 则是让客户端生成一个Connection ID （64 位）来区别不同连接。只要Connection ID 不变，连接就不需要重新建立，即便是客户端的网络发生变化
                • 迁移客户端继续使用相同的会话密钥来加密和解密数据包
                • QUIC 还提供了迁移客户端的自动加密验证
